# PrismWave 开发文档

## 1. 项目定位

**项目名称：PrismWave**

PrismWave 是一个面向 Windows 桌面的玻璃拟态音乐播放器，当前版本以 **本地音乐播放** 为核心，重点放在以下几个方向：

1. UI 统一、简洁、现代，主界面与 FullPlay 页面都保持一致的玻璃拟态视觉。
2. 播放逻辑明确且稳定，所有页面都遵循同一套列表播放 / 单曲循环 / 随机播放规则。
3. 支持主流本地音频格式，兼顾歌词、封面、收藏、专辑 / 艺术家分类等媒体库体验。
4. 在 Windows 上提供兼容模式、WASAPI 共享模式、WASAPI 独占模式三种输出方式。

当前版本已经不是纯概念设计，而是已经有可运行的第一版 demo。

## 2. 当前实现状态

### 2.1 已完成能力

截至当前版本，PrismWave 已完成以下能力：

1. **无边框主窗口**
   - 自绘顶部栏。
   - 放大 / 缩小 / 关闭按钮内建到窗口内。
   - 主界面与顶部栏使用一致的玻璃模糊层级。

2. **主内容结构**
   - 左侧导航：`库`、`专辑`、`艺术家`、`我最爱的`。
   - 主内容列表支持搜索、切换页面、进入专辑详情页、进入艺术家详情页。
   - 专辑和艺术家详情都采用独立页面区块，而不是弹窗样式。

3. **本地音乐库**
   - 支持添加多个本地音乐文件夹。
   - 支持重新扫描。
   - 支持扫描常见音频格式，包括 `mp3 / aac / wav / flac / ogg / ape / dsf / dff / m4a / mp4`。
   - 音乐库、专辑、艺术家、收藏页共用同一套 Track 数据。

4. **播放控制**
   - 播放 / 暂停。
   - 上一首 / 下一首。
   - 进度条拖动。
   - 音量调节。
   - 底部播放栏与 FullPlay 页面共用同一套播放状态。

5. **播放模式逻辑**
   - `列表播放`：手动切歌和自然播放结束都按当前歌单顺序切换。
   - `单曲循环`：手动切歌按当前歌单顺序；自然播放结束时从当前歌曲重新播放。
   - `随机播放`：手动切歌和自然播放结束都在当前歌单内随机切换，且尽量避免命中当前歌曲。
   - 当歌单只有一首歌时，无论哪种模式，手动切歌或自然播放结束都会重新播放当前歌曲。

6. **播放上下文**
   - 在音乐库、收藏、专辑详情、艺术家详情中播放时，切歌逻辑跟随当前歌单上下文。
   - 在音乐库搜索结果中点播歌曲时，播放上下文仍回到完整音乐库顺序，而不是被困在当前搜索结果里。

7. **歌词能力**
   - 支持读取本地歌词。
   - 支持在线歌词获取。
   - 支持在线歌词搜索并手动选择结果。
   - 在线歌词会缓存到本地，后续优先使用缓存。
   - 如果本地歌词不存在，会自动尝试在线歌词。
   - 每首歌的歌词来源选择会被记住，下次播放该歌曲时会恢复上次选择的来源。

8. **FullPlay 页面**
   - 从底部弹出。
   - 独立页面，有自己的顶部栏控制按钮。
   - 左侧显示封面、歌名、歌手、进度与控制按钮。
   - 右侧显示完整歌词列表。
   - 当前歌词居中显示。
   - 非当前歌词保持模糊，当前歌词清晰。
   - 支持逐字高亮。
   - 支持切换歌词时的平滑滚动。

9. **设置页**
   - 语言切换：中英文。
   - 顶部栏空闲显示模式切换。
   - 自定义顶部栏空闲文字。
   - 在线一言获取。
   - 音频输出模式切换。
   - 开发者模式开关。
   - 添加 / 移除 / 重新扫描音乐文件夹。

10. **顶部栏歌词 / 一言显示**
    - 播放时显示当前歌词。
    - 暂停或空闲时显示自定义文字或在线一言。
    - 一言在空闲状态下每 10 秒切换一次。
    - 歌词切换带有类似翻页的过渡效果。

11. **右键菜单与详细信息页**
    - 在歌曲上右键可执行：
      - 收藏 / 取消收藏
      - 定位到文件资源管理器
      - 查看详细信息
    - 详细信息页显示：
      - 封面
      - 歌名
      - 歌手
      - 时长
      - 音轨
      - 码率
      - 采样率
      - 绝对路径

12. **开发者模式**
    - 可弹出独立终端窗口实时输出播放日志。
    - 日志同步写入本地文件。
    - 可用于观察播放状态、输出模式切换、异常恢复等细节。

### 2.2 当前技术实现

当前代码实现并不是文档最早设想的 Flutter + Rust FFI 方案，而是下面这套：

1. **UI**
   - Flutter 桌面端。
   - Riverpod 管理状态。

2. **播放引擎**
   - `just_audio`
   - `just_audio_media_kit`
   - Windows 音频后端通过 `media_kit` 走原生输出。

3. **输出模式**
   - `兼容模式（MPV 默认输出）`
   - `WASAPI Shared`
   - `WASAPI Exclusive`

4. **媒体库与元数据**
   - Dart 扫描本地目录。
   - `metadata_god` 读取元数据与封面。
   - 补充自定义文件头解析以读取部分采样率信息。

5. **歌词系统**
   - 本地歌词解析。
   - 在线歌词拉取与缓存。
   - 手动搜索在线歌词。

### 2.3 当前已知边界

1. 当前仍以本地音乐库为主，未正式支持远程音乐源。
2. 当前详细信息页中的部分参数在少数格式下可能显示为 `--`，取决于元数据和文件头能否成功解析。
3. 目前的播放引擎已经支持 WASAPI 共享和独占，但还没有自研原生解码核心。

## 4. 播放逻辑基线

播放逻辑是当前项目最核心的行为约束，所有页面都必须遵守。

### 4.1 基本规则

1. **播放必须绑定歌单上下文**
   - 点播某首歌时，必须记录其所在歌单。
   - 后续上一首 / 下一首 / 自动切歌都基于这个歌单上下文。

2. **手动切歌与自然结束切歌必须分开处理**
   - 手动点击上一首 / 下一首，是用户主动换歌。
   - 自然播放结束，是自动切歌。
   - 单曲循环模式下，只有自然播放结束会重播当前歌曲，手动切歌仍按歌单顺序切换。

3. **随机模式基于当前歌单**
   - 不能跳出当前歌单。
   - 尽量不随机到当前歌曲。
   - 若歌单长度为 1，则仍播放当前歌曲。

### 4.2 当前实际行为

1. `库`
   - 使用完整音乐库作为播放上下文。

2. `我最爱的`
   - 使用当前收藏歌单作为播放上下文。

3. `专辑详情`
   - 使用当前专辑内歌曲列表作为播放上下文。

4. `艺术家详情`
   - 使用当前艺术家歌曲列表作为播放上下文。

5. `音乐库搜索结果点播`
   - 播放上下文仍为完整音乐库，而不是当前筛选结果。

## 5. UI 设计基线

### 5.1 视觉方向

1. 整体风格：玻璃拟态。
2. 背景：透明根背景 + 半透明渐变层，确保系统毛玻璃可见。
3. 顶部栏与主体模糊强度保持一致。
4. 不使用 Windows 原生标题栏。
5. 控件保持简约，不使用多余边框和蓝色描边。

### 5.2 主页面

1. 左侧导航栏顶部延伸至页面上方。
2. 左上角显示 `PrismWave`。
3. 顶部栏右侧显示歌词 / 一言信息框。
4. 主内容区更贴近顶部栏，减少割裂感。

### 5.3 播放栏

1. 左侧显示封面、歌名、歌手、时长。
2. 中间显示切歌、播放 / 暂停、播放模式按钮。
3. 进度条与音量条使用白色风格。
4. 当前风格以克制、简洁为主。

### 5.4 FullPlay 页面

1. 从底部弹出。
2. 左侧约占 2/5，显示封面、标题、控制栏。
3. 右侧约占 3/5，显示歌词。
4. 不显示主页面顶部歌词栏。
5. 音量滑条默认收起，点击按钮后在右侧展开。

## 6. 音频实现说明

### 6.1 当前实现链路

当前版本的音频处理和播放流程如下：

1. 用户在任意歌单中点选歌曲。
2. UI 将当前歌曲和对应歌单传给 `PlaybackController`。
3. `PlaybackController` 根据当前歌单和播放模式建立播放上下文。
4. 根据设置的输出模式重建或复用播放器后端。
5. 将目标音频文件交给 `just_audio + media_kit`。
6. 在 Windows 上由底层原生输出后端完成音频输出。

### 6.2 输出模式说明

1. **兼容模式**
   - 由 MPV / media_kit 默认后端处理。
   - 兼容性最好。

2. **WASAPI Shared**
   - 使用 Windows 共享模式。
   - 可与其他程序同时发声。

3. **WASAPI Exclusive**
   - 优先使用 Windows 独占模式。
   - 失败时回退共享模式。

### 6.3 当前原则

1. 不做无意义的额外处理链。
2. 保留独占模式优先级，满足高质量本地播放需求。
3. 通过开发者模式日志明确输出后端状态。

### 8.1 本地播放器方向

3. 如果未来要做更彻底的无损链路控制、设备能力探测、DSD 更精细支持，可以重新推进这一层。

## 9. Subsonic / OpenSubsonic 实现方案

这一部分为未来规划，当前版本尚未落地。

### 9.1 目标

将“添加歌曲文件夹”升级为“添加音乐源”，支持两类音乐源：

1. **本地文件夹**
2. **Subsonic / OpenSubsonic 音乐源**

所有音乐源的歌曲最终都进入同一个库列表中展示，并共享：

1. 音乐库
2. 专辑
3. 艺术家
4. 收藏
5. 搜索
6. 播放上下文逻辑

### 9.2 为什么可行

Subsonic / OpenSubsonic 已提供成熟的 REST API，可覆盖：

1. 鉴权
2. 拉取艺术家 / 专辑 / 歌曲
3. 搜索
4. 拉取封面
5. 播放流
6. 收藏 / 取消收藏

可参考：

1. Subsonic API 官方文档  
   https://www.subsonic.org/pages/api.jsp

2. OpenSubsonic 扩展说明  
   https://github.com/owncloud/music/wiki/OpenSubsonic-API

### 9.3 设置页改造方案

将现有设置页中的 `添加歌曲文件夹` 改为 `添加音乐源`。

点击后给出两种入口：

1. `添加本地文件夹`
2. `添加 Subsonic 源`

添加 Subsonic 源时，进入单独的配置页面，建议字段如下：

1. 服务器名称
2. 服务器地址
3. 用户名
4. 密码或 API Key
5. 认证方式
6. 可选音乐文件夹 ID
7. 测试连接
8. 保存

### 9.4 数据模型改造

当前 `Track` 模型以本地文件为中心，不足以同时表示本地与远程歌曲。未来需要引入统一的 `MusicSource` 与增强版 `Track`。

建议新增：

```dart
class MusicSource {
  final String id;
  final MusicSourceType type; // localFolder | subsonic
  final String name;
}
```

远程源需要额外配置：

1. `baseUrl`
2. `username`
3. `authMode`
4. `apiKey` 或密码信息
5. `musicFolderId`

Track 建议增加：

1. `sourceId`
2. `sourceType`
3. `mediaId`
4. `coverArtId`
5. `suffix`
6. `bitRate`
7. `sampleRate`
8. `isRemote`

### 9.5 库同步策略

第一版建议采用 **元数据全量同步 + 本地缓存**，而不是按需懒加载。

原因：

1. 现有 UI 与状态结构默认手里已有完整 `tracks` 列表。
2. 第一版目标是尽快让远程歌曲与本地歌曲在同一套页面中工作。
3. 先做全量同步更容易保证：
   - 搜索统一
   - 专辑统一
   - 艺术家统一
   - 收藏统一

后续若大库性能不足，再引入增量同步或分页。

### 9.6 播放层改造

当前播放层默认使用本地文件路径播放。未来支持 Subsonic 后，必须在播放层区分：

1. **本地歌曲**
   - 继续走本地文件链路。

2. **远程歌曲**
   - 根据 `mediaId + sourceConfig` 生成 `stream` URL。
   - 播放器改为加载远程 URL，而不是本地文件路径。

也就是说，播放层要从“文件路径驱动”升级为“媒体源驱动”。

### 9.7 封面与缓存

远程音乐封面建议使用：

1. `getCoverArt`
2. 本地磁盘缓存缩略图

这样列表滚动时不必频繁请求服务器。

### 9.8 收藏统一

建议未来收藏逻辑分两层：

1. **本地歌曲**
   - 继续使用本地收藏记录。

2. **Subsonic 歌曲**
   - 调用 `star / unstar`

然后在 `我最爱的` 页面统一展示本地收藏与远程收藏。

### 9.9 歌词能力延展

接入 Subsonic 后，未来歌词来源理论上可扩展为：

1. 本地歌词
2. 服务器歌词
3. 在线歌词

不过这不是第一期重点。第一期应先确保：

1. 库同步稳定
2. 播放稳定
3. 封面稳定
4. 收藏稳定

### 9.10 实施阶段建议

建议按四个阶段推进：

#### 阶段 1：数据与设置基础

1. 新增 `MusicSource`
2. 设置页支持添加本地 / Subsonic 两类源
3. 保存远程源配置
4. 测试连接

#### 阶段 2：远程媒体库接入

1. 拉取远程艺术家 / 专辑 / 歌曲
2. 合并进本地总库
3. 支持统一搜索与列表展示
4. 增加封面缓存

#### 阶段 3：远程播放

1. 播放器支持远程流
2. 保持现有播放模式逻辑不变
3. 收藏支持远程 `star / unstar`
4. 开发者模式增加远程流日志

#### 阶段 4：体验增强

1. 增量同步
2. 服务器歌词
3. 多源管理
4. 离线缓存策略

### 9.11 注意事项

1. `定位到文件资源管理器` 不适用于远程歌曲。
2. 远程音频是否真正无损，不只取决于客户端，也取决于服务器是否返回原始码流。
3. 远程认证信息不应只存放在普通偏好存储中，后续建议迁移到更安全的存储方案。

---

## 10. 当前结论

PrismWave 当前已经完成了本地桌面音乐播放器的第一版，并且具备：

1. 稳定的本地播放能力
2. 明确统一的播放模式逻辑
3. 完整的玻璃拟态 UI 基础
4. 歌词、本地库、详细信息、开发者模式等支撑能力

后续如果要继续扩展，最有价值的新方向之一就是：

**在保持现有本地体验不退化的前提下，接入 Subsonic / OpenSubsonic 音乐源，形成本地 + 自托管远程库的统一播放器。**
