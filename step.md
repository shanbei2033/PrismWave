# PrismWave 开发流程（step）

本文将 `dev.md` 的方案落地为可执行的工程流程，按阶段推进，并在每一阶段设置明确的输入、输出与验收标准。  
技术主线保持不变：`Flutter（UI/状态） + Rust（音频核心） + FFI（桥接）`。

## 0. 流程总览

1. 需求冻结与技术基线确认  
2. 工程初始化与架构骨架搭建  
3. 音频核心最小链路打通（Rust + WASAPI）  
4. Flutter 播放链路与状态模型闭环  
5. 媒体库与索引系统落地  
6. 玻璃拟态 UI 与交互完善  
7. DSD/设备切换/错误恢复增强  
8. 性能优化、测试回归与发布

---

## 1. 需求冻结与技术基线确认（第 1 周）

### 目标

- 明确 V1 范围与非目标，避免开发中途范围漂移。
- 冻结核心技术栈、目录结构、接口边界。

### 任务

- 从 `dev.md` 提取并确认 V1 功能：
  - 本地扫描、媒体库（Library/Artists/Albums/Favorites）
  - 播放控制、播放模式、设备切换
  - 玻璃拟态 UI（浅/深色主题）
- 确认非目标：
  - 在线流媒体、账号系统、云同步、社交
- 定义统一术语与事件命名（例如：`onEnded`、`onBuffering`、`setOutputDevice`）。
- 明确目录结构与模块负责人。

### 输出物

- `scope.md`（范围/非目标）
- `architecture.md`（分层与边界）
- `interface-contract.md`（Dart <-> Rust FFI 契约）

### 验收标准

- 团队对 V1 范围无歧义。
- FFI 接口命名与数据结构评审通过。

---

## 2. 工程初始化与架构骨架（第 1~2 周）

### 目标

- 建立可持续迭代的工程底座。

### 任务

- 初始化 Flutter Windows 工程与 Rust Core 工程。
- 接入 `Riverpod`，建立基础状态模块：
  - `PlaybackState`
  - 媒体库状态（扫描、索引、筛选）
- 建立页面壳层：
  - Sidebar、内容区、Now Playing、底部控制栏
- 配置基础日志、错误上报、调试开关。

### 输出物

- 可运行的桌面壳应用（空数据也可启动）。
- Rust 动态库可被 Flutter 成功加载（至少有 `ping`/`version` 级别接口）。

### 验收标准

- Windows 本地可一键启动调试。
- 基础导航、状态更新链路可观察（日志可追踪）。

---

## 3. 音频核心最小链路（第 2~3 周）

### 目标

- 打通 `File -> Decoder -> PCM -> WASAPI` 最小可用播放链路。

### 任务

- Rust 侧实现：
  - 解码 `wav/mp3`（MVP）
  - WASAPI Shared 输出
  - 基础播放控制：`load/play/pause/seek/setVolume`
- 实现关键回调：
  - `onPosition`
  - `onEnded`
  - `onError`
- 设计错误码体系（加载失败、设备不可用、格式不支持、初始化失败）。

### 输出物

- 可稳定播放 wav/mp3 的 Rust 核心。
- Flutter 能通过 FFI 控制播放并接收事件回传。

### 验收标准

- 首次点播可出声，暂停/继续/进度拖动可用。
- 播放结束自动触发 `onEnded`，无崩溃。

---

## 4. 播放行为闭环与列表上下文（第 3~4 周）

### 目标

- 确保切歌逻辑全局一致，避免“入口不同导致行为不同”。

### 任务

- 在 Domain 层固化唯一建链入口：
  - `playFromPlaylist(track, playlist)`
- 完成模式逻辑：
  - `loop`：线性切歌 + 边界回环
  - `single`：自然结束单曲循环，手动切歌同 loop
  - `shuffle`：随机且避免命中当前索引
- 统一 `next/previous/onTrackEnded` 只依赖：
  - `currentPlaylist + currentIndex + playbackMode`

### 输出物

- 播放行为状态机文档（建议补一张流程图）。
- 可复用的播放调度服务（Dart Domain 层）。

### 验收标准

- 任意入口点击歌曲，后续切歌行为一致。
- 搜索过滤后，只在过滤结果内切歌。

---

## 5. 媒体库与索引系统（第 4~6 周）

### 目标

- 建立可支撑 10k 曲目规模的本地媒体管理能力。

### 任务

- 目录扫描与增量更新（Dart Isolate，避免阻塞 UI）。
- 元数据读取：标题、艺术家、专辑、时长、封面。
- 分类视图构建：
  - Library
  - Artists
  - Albums
  - Favorites
- 索引持久化与封面缓存策略。

### 输出物

- 首次扫描与增量扫描能力。
- 基础搜索和筛选能力。

### 验收标准

- 10k 曲目库可在目标时间内完成可交互启动。
- 页面滚动与筛选操作无明显卡顿。

---

## 6. 玻璃拟态 UI 与交互打磨（第 5~7 周，并行）

### 目标

- 在保证性能的前提下，形成统一、克制、现代的视觉体验。

### 任务

- 统一主题变量（blur、alpha、accent、边框、阴影）。
- 落地核心组件玻璃风格：
  - 侧边栏、控制栏、信息卡片、封面卡
- 交互动效：
  - Hover 抬升（2~4px）
  - 动效时长 180~260ms，`easeOutCubic`
- 实现“低特效模式”：
  - 降 blur
  - 减阴影
  - 关闭复杂背景动效

### 输出物

- 浅色/深色双主题。
- 高特效与低特效两档体验。

### 验收标准

- 核心页面视觉一致，无风格割裂。
- 低特效模式可明显改善低配机器帧率。

---

## 7. 格式扩展、DSD 与设备能力增强（第 6~8 周）

### 目标

- 完成 V1 格式能力并增强设备稳定性。

### 任务

- 扩展解码格式：
  - `aac/flac/ogg/ape`
  - `dsf/dff`（DSD）
- DSD 策略：
  - 优先 DoP
  - 不支持时回退 DSD->PCM
  - UI 显示当前链路（`DSD(DoP)` / `DSD->PCM`）
- 输出路径：
  - Shared 默认
  - Exclusive 可选，失败自动回退 Shared
- 设备管理：
  - 监听默认设备变化与热插拔
  - 自动重建会话并恢复播放

### 输出物

- 全格式播放能力（V1 范围内）。
- 设备切换与恢复策略实现。

### 验收标准

- 常见设备切换场景不丢失播放状态。
- Exclusive 失败时可自动回退并提示用户。

---

## 8. 性能优化、测试回归与发布（第 8~10 周）

### 目标

- 达成 V1 性能指标并具备可发布质量。

### 任务

- 对照目标指标做压测与调优：
  - 冷启动 <= 2.5s（10k 曲目库）
  - 首次点播 <= 180ms
  - 相邻切歌 <= 120ms
  - 主界面滚动 60fps
  - CPU：空闲 <= 3%，播放 <= 10%
- 完成测试矩阵：
  - 单元测试（模式算法、索引边界、元数据解析）
  - 集成测试（FFI 闭环、设备切换、DSD 模式切换）
  - 手工回归（入口一致性、搜索后切歌范围、主题可读性、低特效）
- 完成打包、安装流程、发布说明、日志诊断导出。

### 输出物

- 回归测试报告
- 性能报告
- 发布包与 Release Note

### 验收标准

- P0/P1 问题清零。
- 所有核心指标达标或有书面豁免说明。

---

## 9. 质量闸门（每阶段通用）

每个阶段结束前必须通过以下闸门：

1. 功能闸门：本阶段定义的核心能力可演示、可复现。  
2. 稳定闸门：无阻断崩溃；错误有可读提示与日志。  
3. 性能闸门：关键路径无明显退化。  
4. 文档闸门：接口、行为、已知问题有记录。  
5. 回归闸门：历史关键用例通过。

---

## 10. 建议执行节奏（周节奏）

1. 周一：拆分任务、确认风险、冻结本周目标。  
2. 周二到周四：开发 + 日常联调 + 小步合并。  
3. 周五：阶段验收、回归测试、文档更新、下周准备。

建议保持“小步提交 + 每周可演示 + 双周里程碑复盘”的节奏，确保 PrismWave 在视觉质量与音频稳定性上同步推进。
